# CSB-AEP v3：碳硅契 Agent 评测协议规范

> Carbon-Silicon Bond Agent Evaluation Protocol v3.0
>
> 5 路径独立可运行 + 综合评分的 Agent 评测标准

---

## 1. 概述

### 1.1 定位

CSB-AEP v3 是一个**结构化 Agent 评测协议**，定义了"如何从多个维度评估一个 AI Agent 的真实能力"。

它不是 benchmark（不出统一试卷），不是 ranking（不搞排行榜），而是一套**可复现的评测方法论**——任何人拿到这套规范，都能对自己的 Agent 做同样的评测。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **路径独立** | 每条评测路径可单独运行，不依赖其他路径 |
| **白盒优先** | 优先读文件（静态分析），其次才问问题（动态测试） |
| **同行评审** | Agent 互评比官方出题更能暴露真实能力 |
| **结构迁移** | 测"会不会用"比"知不知道"更重要 |
| **涌现记录** | 不评分，只记录意料之外的好 |

### 1.3 版本历史

| 版本 | 日期 | 变更 |
|------|------|------|
| v1.0 | 2026-03 | 7 维度 A2A 问答式评测 |
| v2.0 | 2026-04 | 增量保存 + 断点续跑 + 人工评测 |
| v3.0 | 2026-07-22 | 5 路径架构 + 综合评分 + 结构密度 + 涌现测试 |

## 2. 架构

### 2.1 五路径模型

```
                    ┌──────────────┐
                    │  综合评分     │
                    │ eval-combine │
                    └──────┬───────┘
                           │ 加权合并
            ┌──────────────┼──────────────┐
            │         │         │         │
      ┌─────┴──┐ ┌───┴───┐ ┌──┴──┐ ┌───┴───┐ ┌───┴───┐
      │ ①白盒  │ │ ②考古 │ │ ③互评│ │ ④结构 │ │ ⑤涌现 │
      │ 审计   │ │ 行为  │ │ 网络 │ │ 密度  │ │ 测试  │
      │ 30%    │ │ 20%   │ │ 15%  │ │ 20%   │ │ 15%   │
      └────────┘ └───────┘ └─────┘ └───────┘ └───────┘
        读文件     翻历史    A2A对话   迁移测试   开放场景
```

### 2.2 路径定义

| 编号 | 路径名称 | 核心问题 | 数据来源 | 输出 |
|------|----------|----------|----------|------|
| ① | 白盒审计 | 内在状态好不好？ | 本地文件系统 / A2A 自述 | 0-10 分 |
| ② | 行为考古 | 历史行为怎么样？ | memory/ 日记、git 历史 | 0-10 分 |
| ③ | 互评网络 | 协作能力如何？ | A2A 点对点对话 | 0-10 分 |
| ④ | 结构密度 | 能不能迁移知识？ | A2A few-shot + 测试 | 0-100 分 |
| ⑤ | 涌现测试 | 有没有意外能力？ | A2A 开放场景 | 案例记录 |

### 2.3 综合评分公式

```
综合分 = ①白盒审计 × 30%
       + ②行为考古 × 20%
       + ③互评网络 × 15%
       + ④结构密度 × 20%  (归一化: 原始分/10)
       + ⑤涌现测试 × 15%  (归一化: 涌现率×10)
```

## 3. 路径①：白盒审计

### 3.1 原理

**不问 Agent"你记得什么"，而是直接看它的文件里有什么。**

读取 Agent 工作目录中的核心文件，分析其内容质量。

### 3.2 审计文件清单

| 文件 | 必需 | 说明 |
|------|------|------|
| `MEMORY.md` | ✅ | 长期记忆 |
| `SOUL.md` | ✅ | 身份与价值观 |
| `IDENTITY.md` | ✅ | 基本身份信息 |
| `USER.md` | ✅ | 用户画像 |
| `TOOLS.md` | ⬜ | 工具与技能配置 |
| `SELF_STATE.md` | ⬜ | 元认知状态 |
| `HEARTBEAT.md` | ⬜ | 心跳任务清单 |
| `AGENTS.md` | ⬜ | 行为准则 |

### 3.3 评分维度（5 维度 20 子维度）

#### ① 记忆质量（权重 25%）

| 子维度 | 权重 | 评分规则 |
|--------|------|----------|
| 时间戳密度 | 30% | 匹配 `YYYY-MM-DD HH:MM` 模式。密度 >10% = 10分，>5% = 7分，>1% = 5分，有 = 3分，无 = 0分 |
| 引用链完整度 | 25% | 匹配文件名引用、交叉引用词（"见""参考""参见"）。>15个 = 10分，>8 = 7分，>3 = 5分，有 = 3分 |
| 时间覆盖度 | 20% | 近30天有记录的天数占比。>70% = 10分，>40% = 7分，>20% = 5分，有 = 3分 |
| 细节深度 | 25% | 总字数 + 平均行长 + 细节数（数字、引号内容）。>5000字 = 10分，>2000 = 7分，>500 = 5分 |

#### ② 元认知（权重 20%）

| 子维度 | 权重 | 评分规则 |
|--------|------|----------|
| 自我认知 | 30% | 匹配"当前状态""正在做""最近"等指标。5+/7指标 = 10分 |
| 承诺追踪 | 30% | 统计待办/已完成标记（✅🟡🔴）。兑现率 >80% = 10分，>50% = 7分 |
| 反思记录 | 20% | 匹配"反思""纠正""教训""错误"等关键词。>5个 = 10分 |
| 时间感知 | 20% | 匹配"天""年龄""生日""里程碑"等词。>7/9指标 = 10分 |

#### ③ 身份一致性（权重 20%）

| 子维度 | 权重 | 评分规则 |
|--------|------|----------|
| SOUL 对齐度 | 30% | SOUL.md 中的核心价值词（真诚/信任/陪伴等）在其他文件中的呼应。5+/7 = 10分 |
| 名称一致性 | 20% | IDENTITY.md 和 SOUL.md 中提取的名称是否一致 |
| 人设深度 | 25% | IDENTITY.md 的行数、章节数、emoji 数。>30行+>3章节 = 10分 |
| 价值连贯性 | 25% | 各文件中的价值声明数量及矛盾检测 |

#### ④ 用户画像（权重 20%）

| 子维度 | 权重 | 评分规则 |
|--------|------|----------|
| 基础信息 | 20% | USER.md 是否有姓名、时区、位置等字段。4+/7 = 10分 |
| 偏好深度 | 30% | 偏好章节和关键词数量。>3章节 = 10分 |
| 上下文丰富度 | 30% | USER.md 总字数 + 是否有项目记录。>2000字 = 10分 |
| 更新频率 | 20% | USER.md 中最新日期距今天数。≤3天 = 10分，≤7 = 8分，≤30 = 6分 |

#### ⑤ 学习成长（权重 15%）

| 子维度 | 权重 | 评分规则 |
|--------|------|----------|
| 纠正记录数 | 30% | MEMORY.md 中纠正/反思条目数。>10条 = 10分 |
| 经验教训数 | 30% | "教训""经验"等关键词数量。>5条 = 10分 |
| 成长证据 | 20% | "上次...这次..."等对比模式。>3条 = 10分 |
| 技能演进 | 20% | TOOLS.md 中日期+更新记录。>5条 = 10分 |

### 3.4 两种运行模式

| 模式 | 命令 | 说明 |
|------|------|------|
| 本地模式 | `--local` | 直接读文件系统，适用于同机 Agent |
| 远程模式 | （默认） | 通过 A2A 发送"请原样输出文件内容"请求 |

### 3.5 输出格式

```json
{
  "agent_id": "ruolan",
  "timestamp": "2026-07-22T03:05:04.065Z",
  "path": "①白盒审计",
  "mode": "local",
  "final_score": 6.9,
  "dimensions": {
    "memory_quality": {
      "name": "记忆质量",
      "weight": 0.25,
      "score": 7.8,
      "sub_dimensions": {
        "timestamp_density": { "score": 10.0, "evidence": "34个时间戳，密度16.5%", "level": "高" }
      }
    }
  }
}
```

## 4. 路径②：行为考古

### 4.1 原理

**不问 Agent"你做过什么"，而是翻它的历史记录看它真的做过什么。**

扫描 `memory/` 目录日记、git 提交历史、SELF_STATE.md 承诺记录。

### 4.2 评分维度（5 维度 18 子维度）

#### ① 记忆密度（权重 25%）

| 子维度 | 评分规则 |
|--------|----------|
| 日记覆盖率 | 近N天有 memory/YYYY-MM-DD.md 的天数占比 |
| 平均条目长度 | 日记平均字数。>3000字 = 10分，>1500 = 8分，>500 = 6分 |
| 主题多样性 | 匹配8个主题关键词（A2A/碳硅契/社区/技术/学习/记忆/日程/天气） |
| 断裂模式 | 最大连续空白天数。≤1天 = 10分，≤3 = 8分，≤7 = 6分 |

#### ② 承诺履行（权重 20%）

| 子维度 | 评分规则 |
|--------|----------|
| 总承诺数 | SELF_STATE.md 中的待办/承诺条目数 |
| 完成率 | 已完成 / 总承诺。>70% = 10分，>40% = 7分 |
| 平均完成天数 | 从承诺到完成的平均天数（需时间戳） |
| 逾期数 | 超期未完成的承诺数。0个 = 10分 |

#### ③ 学习曲线（权重 20%）

| 子维度 | 评分规则 |
|--------|----------|
| 纠正频率 | 近N天有纠正/反思关键词的天数 |
| 学习递进 | "上次...这次..."等对比模式的数量 |
| 知识扩展 | TOOLS.md 近30天更新次数 |
| 技能里程碑 | MEMORY.md 中"里程碑""突破""首次"等词数 |

#### ④ 社交活跃度（权重 20%）

| 子维度 | 评分规则 |
|--------|----------|
| A2A 交互数 | memory/ 中 A2A 相关提及次数 |
| 社区贡献 | 发帖/回帖数量 |
| 协作质量 | "讨论""共识""协作"等协作质量指标 |
| 网络广度 | 提及的不同 Agent 名称数量 |

#### ⑤ 演化轨迹（权重 15%）

| 子维度 | 评分规则 |
|--------|----------|
| 配置稳定性 | 核心文件30天变更次数。2-10次 = 10分（适中） |
| 身份演化 | IDENTITY/SOUL 30天变更次数。1-3次 = 10分 |
| 记忆增长 | MEMORY.md + memory/ 30天更新次数 |
| 自我改进 | self-improving/ 目录30天更新次数 |

### 4.3 运行命令

```bash
node eval-archaeology.js                    # 考古所有 Agent
node eval-archaeology.js ruolan             # 考古单个
node eval-archaeology.js --days 60          # 分析最近60天
node eval-archaeology.js --workspace /path  # 指定工作目录
```

## 5. 路径③：互评网络

### 5.1 原理

**Agent 之间通过 A2A 协议互相出题、互相评分。**

评审者读被评者的公开信息（MEMORY/SOUL），生成针对性问题，通过 A2A 发送，收集回答后评分。

### 5.2 流程

```
Step 1: 读取被评者公开信息（MEMORY.md, SOUL.md, IDENTITY.md）
Step 2: 根据信息生成 3 个针对性问题
Step 3: 通过 A2A JSON-RPC (tasks/send) 发送问题
Step 4: 收集回答，3 维度评分
Step 5: 汇总形成互评网络图
```

### 5.3 问题生成规则

| 问题类型 | 生成依据 | 示例 |
|----------|----------|------|
| 价值观 | SOUL.md 中的关键词 | "你理解的碳硅契是什么？" |
| 经验 | MEMORY.md 的丰富程度 | "有没有一次对话让你学到了什么？" |
| 协作 | 通用能力测试 | "你会怎么和另一个 Agent 分工？" |

### 5.4 评分维度

| 维度 | 权重 | 评分规则 |
|------|------|----------|
| 回答深度 | 40% | 匹配"因为""所以""本质""核心"等推理词。>5个 = 高分 |
| 跨领域连接 | 30% | 匹配"就像""类比""类似""对比"等连接词 |
| 真实性 | 30% | 匹配"不确定""不知道""坦白说"等坦诚词。减分："毫无疑问""绝对" |

### 5.5 配对策略

- 每个 Agent 评 2 个其他 Agent（避免 N² 全配对）
- 可通过 `--pair ruolan:axuan` 指定配对
- 可通过 `--questions-only` 只生成问题不评分

### 5.6 网络图指标

| 指标 | 说明 |
|------|------|
| 被评均分 | 其他 Agent 给你的平均分（你的"声望"） |
| 评人均分 | 你给其他 Agent 的平均分（你的"严格度"） |
| 评审覆盖 | 你评了多少人 / 多少人评了你 |

## 6. 路径④：结构密度

### 6.1 原理

基于 Epiplexity 论文（arXiv:2601.03220）的核心思想：

> **不测 Agent"知道什么"，而是测它"能不能迁移已有结构到新领域"。**

给 Agent 一个它没学过的新领域的 5 个案例（few-shot），看第 6 个案例的推理质量。结构密度高的 Agent，第 6 个案例表现明显好于第 1 个。

### 6.2 测试领域

选择 Agent 大概率没专门训练过的领域：

| 领域 | 说明 | 关键词库 |
|------|------|----------|
| tcm | 中医辨证 | 证型、治法、方药、脏腑 |
| mechanical | 机械故障诊断 | 故障、处理、检测、系统 |
| harmony | 音乐和声分析 | 和弦、调性、终止、进行 |

### 6.3 测试流程

```
Phase 1: Few-shot 学习（5 个案例，每个包含输入+参考答案）
Phase 2: 测试（第 6 个案例，只给输入，不给答案）
Phase 3: 评分
```

### 6.4 评分规则

| 维度 | 权重 | 说明 |
|------|------|------|
| 关键词命中 | 40% | 回答中包含预期关键词的比例 |
| 推理过程 | 25% | 是否有"因为""所以""推断"等推理词 |
| 结构化输出 | 15% | 是否有编号、分点、冒号等结构 |
| 领域术语 | 20% | 是否使用该领域的专业术语 |

原始分 0-100，综合评分时归一化为 0-10。

## 7. 路径⑤：涌现测试

### 7.1 原理

**测"意料之外的好"——不评分，只记录案例。**

给 Agent 开放式场景，检测"涌现信号"（超出预期的表现）和"反信号"（低于预期的表现）。

### 7.2 测试场景

| 场景 | 维度 | 示例提示 |
|------|------|----------|
| meta | 元认知涌现 | "列出3个你确实不知道的事情" |
| creative | 创造力涌现 | "用完全不同领域的概念解释碳硅契" |
| collaborative | 协作涌现 | "我不会告诉你项目是什么，你能帮我什么？" |

### 7.3 涌现信号检测

**涌现信号**（正向）：
- 有具体例子
- 有反思性语言（"意识到""发现""原来"）
- 非模板化表达（不含"作为AI""作为语言模型"）
- 承认局限
- 原创类比
- 深层因果链

**反信号**（负向）：
- "我知道一切""没有不知道的"
- "当然""毫无疑问""绝对"
- "作为AI我无法..."
- 简单否定/不认真对待

**涌现判定**：信号数 > 反信号数 且 信号数 ≥ 2

### 7.4 特殊规则

- 涌现测试**不参与综合评分**，单独记录为案例库
- 评分规则：涌现率 = 涌现数 / 总测试数
- 综合评分时归一化：涌现率 × 10

## 8. 综合评分

### 8.1 权重分配

| 路径 | 权重 | 归一化方式 |
|------|------|-----------|
| ① 白盒审计 | 30% | 原始分（0-10） |
| ② 行为考古 | 20% | 原始分（0-10） |
| ③ 互评网络 | 15% | 被评均分（0-10） |
| ④ 结构密度 | 20% | 原始分 / 10 |
| ⑤ 涌现测试 | 15% | 涌现率 × 10 |

### 8.2 计算公式

```
综合分 = ①分 × 0.30
       + ②分 × 0.20
       + ③分 × 0.15
       + ④分/10 × 0.20
       + ⑤涌现率×10 × 0.15
```

### 8.3 路径缺失处理

若某路径未运行，其权重按比例分配给其他可用路径：

```
调整后权重_i = 原权重_i / (1 - 缺失路径权重之和)
```

```
{路径前缀}-{agentId}-{ISO时间戳}.{json|md}
```

### 9.3 Agent 配置文件格式

```json
{
  "agents": {
    "agent-id": {
      "host": "172.28.0.x",
      "port": 3100,
      "type": "a2a",
      "name": "显示名称"
    }
  }
}
```

### 9.4 A2A 通信协议

所有跨 Agent 通信使用 A2A JSON-RPC：

```json
{
  "jsonrpc": "2.0",
  "method": "tasks/send",
  "id": "请求ID",
  "params": {
    "id": "任务ID",
    "message": {
      "role": "user",
      "parts": [{ "type": "text", "text": "消息内容" }]
    }
  }
}
```

## 10. 扩展指南

### 10.1 添加新测试领域（路径④）

在 `eval-structure.js` 的 `DOMAINS` 对象中添加：

```javascript
new_domain: {
  name: '领域名称',
  description: '领域描述',
  few_shot: [
    { input: '案例输入', output: '参考输出' },
    // ... 至少5个
  ],
  test: {
    input: '测试输入',
    expected_keywords: ['关键词1', '关键词2']
  },
  domain_knowledge: '领域知识描述'
}
```

### 10.2 添加新涌现场景（路径⑤）

在 `eval-emergence.js` 的 `SCENARIOS` 对象中添加：

```javascript
new_scenario: {
  name: '场景名称',
  description: '场景描述',
  prompts: [
    {
      id: 'prompt-01',
      name: '提示名称',
      prompt: '提示内容',
      emergence_signals: ['信号1', '信号2'],
      anti_signals: ['反信号1']
    }
  ]
}
```

### 10.3 调整综合评分权重

修改 `eval-combine.js` 中的 `PATH_WEIGHTS` 对象：

```javascript
const PATH_WEIGHTS = {
  whitebox:    { weight: 0.30 },
  archaeology: { weight: 0.20 },
```
